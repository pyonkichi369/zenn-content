プログラミングにClaude Codeを使い始めたけど、「何を指示すればいいか分からない」と感じていませんか？

Claude Codeは指示の出し方で出力品質が激変します。曖昧な指示だと曖昧なコードが返ってくる。具体的な指示なら、シニアエンジニアが書いたような正確なコードが返ってくる。

この記事では、私が実務で毎日使っている**即効プロンプト20選**をそのまま公開します。コピペして変数名だけ変えれば、すぐに使えます。

## こんな人向け

- Claude Codeを導入したが活用しきれていない
- 毎回ゼロから指示を考えるのが面倒
- 他の人がどう使っているか知りたい

## 無料パート：プロンプトの「型」

Claude Codeのプロンプトには効果的な型があります。

**型1: コンテキスト + 指示 + 制約**

```
@src/api/users.py のgetUser関数にバリデーションを追加して。
メールアドレスの形式チェックと、存在しないユーザーには404を返す。
既存のエラーハンドリングパターンに合わせて。
```

ポイントは3つ：
1. **@ファイルパス** で対象を明示する
2. **何をするか** を具体的に言う
3. **制約** で品質を担保する

**型2: Before/After指定**

```
この関数のレスポンス時間が200ms超えてる。
50ms以下にして。N+1クエリが原因だと思う。
```

現状と目標を明示すると、Claude Codeは差分だけを正確に修正します。

**型3: 既存コードの参照**

```
@src/api/orders.py と同じパターンで @src/api/payments.py にCRUDを追加して。
テストも @tests/test_orders.py を参考に作って。
```

「このファイルと同じように」が最強の品質指定です。

ここまでが基本の型です。次のセクションでは、これらの型を使った**実務で即使えるプロンプト20選**を紹介します。

## カテゴリ1: バグ修正（5選）

### 1. エラーログからの修正

```
このエラーが出てる:
[エラーメッセージをペースト]

@src/該当ファイル.py を修正して。
根本原因を特定してから直して、同じパターンの箇所が他にあれば一緒に直して。
```

### 2. テスト失敗の修正

```
make test で以下が失敗:
[テスト出力をペースト]

テストが正しい前提で、実装側を修正して。
テストの意図を変えないで。
```

### 3. パフォーマンス劣化の修正

```
@src/api/search.py の検索が遅い（現在800ms、目標100ms以下）。
explain analyzeの結果:
[EXPLAIN結果をペースト]

インデックス追加かクエリ書き換えで対応して。
```

### 4. 型エラーの一括修正

```
mypy --strict で出てるエラーを全部直して。
any型は使わないで、具体的な型を推論して。
```

### 5. 本番障害の緊急修正

```
本番で500エラーが発生中。
ログ:
[ログをペースト]

最小限の変更で修正して。リファクタリングはしないで。
修正後に影響範囲を教えて。
```

## カテゴリ2: 新機能追加（5選）

### 6. CRUD API追加

```
@src/models/product.py のモデルに対してCRUD APIを作って。
場所: @src/api/products.py（新規）
参考: @src/api/users.py と同じパターン
テストも作って: @tests/test_products.py
バリデーション、ページネーション、エラーハンドリング含めて。
```

### 7. 認証・認可の追加

```
@src/api/admin.py の全エンドポイントに認証を追加して。
- 未認証 → 401
- 権限不足 → 403
- @src/core/security.py の既存のauth依存関係を使って
```

### 8. バッチ処理の実装

```
毎日0時に実行するバッチ処理を作って。
処理: 30日以上未ログインのユーザーにリマインドメール送信
要件:
- 1000件ずつチャンク処理
- 失敗しても他のチャンクに影響しない
- 実行ログを残す
```

### 9. WebSocket追加

```
@src/api/chat.py にWebSocketエンドポイントを追加して。
機能: リアルタイムチャット
- 接続管理（接続/切断/再接続）
- メッセージブロードキャスト
- 心拍チェック（30秒間隔）
```

### 10. ファイルアップロード

```
画像アップロードAPIを作って。
- 最大5MB、jpg/png/webpのみ
- S3にアップロード（presigned URL方式）
- サムネイル自動生成（200x200）
- メタデータをDBに保存
```

## カテゴリ3: リファクタリング（5選）

### 11. 巨大ファイルの分割

```
@src/api/orders.py が600行超えてる。
サービス層を抽出して分割して:
- @src/api/orders.py（ルーター、薄く）
- @src/services/orders.py（ビジネスロジック）
外部インターフェースは変えないで。テストも更新して。
```

### 12. 重複コードの統合

```
@src/api/ 以下で重複しているバリデーションロジックを見つけて、
共通ユーティリティに抽出して。
抽出先: @src/core/validators.py
呼び出し元も全部更新して。
```

### 13. エラーハンドリングの統一

```
@src/api/ 全体のエラーハンドリングを統一して。
ルール:
- ビジネスエラー → HTTPException
- 外部API失敗 → リトライ3回 → fallback値
- 予期しないエラー → 500 + 構造化ログ
```

### 14. テストの改善

```
@tests/test_orders.py のテストを改善して。
- モックを減らして統合テストに寄せて
- エッジケースを追加（空リスト、上限値、不正入力）
- テスト名を test_<action>_<condition>_<expected> に統一
```

### 15. 命名の改善

```
@src/services/data.py の命名が曖昧。
- process_data → 具体的な名前に
- handle_result → 具体的な名前に
- tmp, x, data → 意味のある名前に
変数名・関数名を文脈に合った名前にリネームして。
全ファイルの参照も更新して。
```

## カテゴリ4: 調査・理解（3選）

### 16. コード解読

```
@src/engine/rule_engine.py を読んで、以下を教えて:
1. このモジュールの責務（1文で）
2. 主要なパブリック関数とその役割
3. データフロー（入力→処理→出力）
4. 改善すべき点（あれば）
```

### 17. 依存関係の可視化

```
@src/api/ から @src/services/ と @src/models/ への
依存関係をリストアップして。
循環依存があれば指摘して。
```

### 18. セキュリティレビュー

```
@src/api/auth.py をセキュリティ観点でレビューして。
チェック項目:
- SQLインジェクション
- 認証バイパス
- トークン管理
- レート制限
- OWASP Top 10該当項目
```

## カテゴリ5: 自動化・効率化（2選）

### 19. CI/CDパイプライン

```
GitHub Actionsで以下のCIを作って: @.github/workflows/ci.yml
- lint（ruff）
- 型チェック（mypy）
- テスト（pytest --cov）
- セキュリティ（pip audit + detect-secrets）
PR時に自動実行。mainマージ時にデプロイ。
```

### 20. Makefile整備

```
プロジェクトのMakefileを作って。
必要なコマンド:
- make dev（開発サーバー起動）
- make test（テスト実行）
- make lint（リント+自動修正）
- make db-migrate（DBマイグレーション）
- make clean（キャッシュ削除）
各コマンドにヘルプコメントつけて。
```

## 使い方のコツ

1. **@ファイルパス** を必ずつける。Claude Codeはファイルを読んでから修正するので精度が上がる
2. **「参考: @既存ファイル」** が最強。既存コードのパターンを踏襲させると品質が安定する
3. **制約を明示** する。「テストの意図を変えないで」「外部インターフェースは変えないで」など
4. **1プロンプト1タスク** が基本。複数タスクを混ぜると精度が落ちる
5. **CLAUDE.md** にプロジェクト固有のルールを書いておくと、毎回の指示が短くて済む
