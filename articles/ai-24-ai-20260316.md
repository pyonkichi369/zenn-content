## 1. なぜAIエージェントにガバナンスが必要なのか？

### AIエージェントの暴走はどのように起きるのか？

AIエージェントの暴走は、SF映画のような劇的なものではありません。現実には「小さな判断ミスの蓄積」です。

例えば、コンテンツ生成エージェントが不正確な情報を含む記事を50本自動公開する。収益エージェントが利用規約に違反する価格設定を行う。セキュリティエージェントが「安全」と判定したコードに脆弱性がある——こうした事象が同時多発的に起きた時、人間一人では対処しきれません。

### ガバナンスなしの運用で実際に何が起きたか

私が最初にAIエージェントの自律運用を始めた時、ガバナンスは最小限でした。結果として：

- エージェントが生成した記事に事実誤認が含まれ、読者から指摘を受けた
- 自動承認のまま公開された技術記事のコード例にセキュリティホールがあった
- 複数エージェント間で矛盾する判断が出力され、整合性が取れなくなった

これらの経験から、**ガバナンスは「制約」ではなく「品質の土台」**だと確信しました。

### ガバナンスの3つの柱とは？

| 柱 | 目的 | 具体例 |
|---|---|---|
| **品質管理** | 出力の質を担保する | 品質ゲート、レビューワークフロー |
| **リスク制御** | 損害を防止・最小化する | エスカレーション、自動停止 |
| **透明性** | 判断過程を追跡可能にする | 監査証跡、ログ、レポート |

## 2. 4段階意思決定フレームワークとは何か？

AIエージェントの全ての行動を人間が承認するのは非現実的です。かといって全てを自律に任せるのは危険です。この二項対立を解決するのが、**4段階意思決定フレームワーク**です。

### AUTO — 人間の関与なしで実行

```yaml
# AUTO: 低リスク・高確信度・可逆的な操作
auto_rules:
  - name: "コードフォーマット"
    conditions:
      - risk_level: low
      - reversible: true
      - confidence: ">= 0.95"
    action: execute_immediately
    logging: standard
```

AUTOに分類される操作の例：
- コードのフォーマット修正（prettier、ruff）
- テストの実行と結果レポート
- 日次ステータスレポートの生成
- 既存記事のメタデータ更新

**判定基準**: リスクが低く、失敗しても即座に元に戻せる操作のみ。

### NOTIFY — 実行後に通知

```yaml
# NOTIFY: 中リスク・可逆的・実行後に日次サマリーで通知
notify_rules:
  - name: "コンテンツ公開"
    conditions:
      - risk_level: medium
      - quality_score: ">= 7"
      - passed_review: true
    action: execute_and_notify
    notification_frequency: daily
```

NOTIFYの操作例：
- 品質ゲート通過済みの記事公開
- 依存パッケージのマイナーバージョン更新
- パフォーマンス最適化の適用
- 既存機能のリファクタリング

### CONFIRM — 人間の事前承認が必要

```yaml
# CONFIRM: 高リスク・不可逆的な可能性がある操作
confirm_rules:
  - name: "有料商品の価格変更"
    conditions:
      - risk_level: high
      - involves_money: true
    action: require_human_approval
    timeout_hours: 24
    default_on_timeout: reject  # タイムアウト時は拒否
```

CONFIRMが必要な操作：
- 有料コンテンツの価格設定・変更
- 新しい外部APIとの連携開始
- データベーススキーマの変更
- 新しいエージェントの追加

### HALT — 即座に停止・人間にエスカレーション

```yaml
# HALT: セキュリティ違反・マニフェスト違反
halt_rules:
  - name: "セキュリティ違反検知"
    conditions:
      - security_violation: true
    action: immediate_stop
    notification: immediate
    logging: forensic  # 完全な証跡を記録
```

HALTが発動する条件：
- シークレット（APIキー、パスワード）のハードコード検知
- マニフェスト（不変ルール）への違反
- 認可されていない外部通信の検出
- 予算上限の超過

## 3. 品質ゲートはどのように設計するのか？

### 8ステップ品質バリデーションとは

AEGISでは、全てのエージェント出力が8ステップの品質ゲートを通過します。

```python
# engine/common/quality_gate.py
QUALITY_GATES = [
    ("syntax",        "構文チェック — パーサーによる検証"),
    ("types",         "型互換性 — 型チェッカーによる検証"),
    ("lint",          "品質チェック — リンターによる検証"),
    ("security",      "セキュリティ — OWASP Top 10チェック"),
    ("tests",         "テスト — ユニット>=80%, 統合>=70%"),
    ("performance",   "パフォーマンス — ベンチマーク基準"),
    ("documentation", "ドキュメント — 必須項目の存在確認"),
    ("integration",   "統合テスト — E2Eシナリオ"),
]

def run_quality_gate(output, gate_level="standard"):
    """出力を品質ゲートに通す"""
    results = []
    for gate_name, description in QUALITY_GATES:
        result = execute_gate(gate_name, output)
        results.append(result)

        # CRITICALゲートが失敗したら即停止
        if result.severity == "CRITICAL" and not result.passed:
            return QualityResult(
                passed=False,
                failed_gate=gate_name,
                results=results,
                action="HALT"
            )

    score = sum(r.score for r in results) / len(results)
    return QualityResult(
        passed=score >= 7.0,
        score=score,
        results=results,
        action="PASS" if score >= 7.0 else "REVIEW"
    )
```

### なぜ7.0がスコア閾値なのか？

スコア閾値の設定は難しい問題です。高すぎればエージェントの効率が落ち、低すぎれば品質が保てません。

実運用データから：
- **スコア9.0以上**: 全出力の約15%。完璧だが、到達に時間がかかりすぎる
- **スコア7.0-8.9**: 全出力の約60%。実用上十分な品質
- **スコア5.0-6.9**: 全出力の約20%。修正すれば使える
- **スコア5.0未満**: 全出力の約5%。根本的なやり直しが必要

7.0を閾値にすることで、**品質と効率のバランス**が最適化されます。

### コンテンツ専用の品質ゲート

コードだけでなく、コンテンツ（記事、ドキュメント）にも品質ゲートを設けています。


## 4. レッドチームはどう運用するのか？

### レッドチームとは何か？

レッドチームとは、**意図的に反対意見を述べる役割**を持つエージェント群です。AEGISでは2体のレッドチームエージェントが常駐しています。

| エージェント | 役割 | 問いかけの例 |
|---|---|---|
| **DevilsAdvocate** | 仮定を疑い、隠れたリスクを発見する | 「この前提が間違っていたら？」 |
| **Skeptic** | 代替案を探り、事前検死を行う | 「もっと良い方法はないか？」 |

### レッドチームはいつ介入するのか？

全ての意思決定にレッドチームが介入すると、効率が壊滅的に低下します。そのため、**リスクレベルに応じた介入ルール**を設定しています。

```yaml
# workflows/mandatory_review.yaml
review_tiers:
  OPERATIONAL:
    reviewers: 1
    time_limit: "5min"
    red_team: false      # レッドチーム不要

  TACTICAL:
    reviewers: 1
    time_limit: "15min"
    red_team: false       # レッドチーム不要

  STRATEGIC:
    reviewers: 2+
    time_limit: "30min"
    red_team: true        # レッドチーム必須

  CRITICAL:
    reviewers: 3+
    time_limit: "60min"
    red_team: true        # レッドチーム + 完全な議事録
    human_approval: required
```

**STRATEGIC以上の意思決定には必ずレッドチームが参加します。**

### レッドチームの効果を最大化する方法

レッドチームが効果を発揮するための3つの原則：

1. **独立性**: レッドチームは提案者の影響を受けてはいけない。別のコンテキストで独立して分析する
2. **構造化された反論**: 「なんとなく不安」ではなく、具体的な反論をフレームワークに沿って出す
3. **記録と追跡**: レッドチームの指摘が無視された場合、その判断の根拠も記録する

```python
# レッドチーム出力の構造
red_team_report = {
    "proposal_id": "PROP-2026-0315",
    "risk_assessment": {
        "identified_risks": [
            {
                "risk": "外部APIのレート制限超過",
                "probability": "HIGH",
                "impact": "MEDIUM",
                "mitigation": "サーキットブレーカーの実装"
            }
        ],
        "hidden_assumptions": [
            "APIの可用性が99.9%であると仮定しているが、実測値は98.2%"
        ],
        "alternative_approaches": [
            "ローカルキャッシュ + 非同期同期で外部依存を削減"
        ]
    },
    "verdict": "FLAG",  # PASS / FLAG / BLOCK / ESCALATE / HALT
    "confidence": 0.85
}
```

## 5. エスカレーションはどう実装するのか？

### 信頼度に基づくスペクトラム

全てのエージェント判断には信頼度スコアが付与されます。このスコアに基づいて、自動実行か人間エスカレーションかが決まります。

```python
def determine_escalation(action, confidence, risk_level):
    """信頼度とリスクに基づくエスカレーション判定"""

    # HALT条件: セキュリティ違反は信頼度に関係なく即停止
    if action.involves_security_violation:
        return "HALT"

    # リスクマトリクス
    if risk_level == "critical":
        return "CONFIRM"  # 常に人間承認

    if risk_level == "high":
        if confidence >= 0.95:
            return "NOTIFY"
        return "CONFIRM"

    if risk_level == "medium":
        if confidence >= 0.80:
            return "AUTO"
        return "NOTIFY"

    # low risk
    return "AUTO"
```

### 信頼スコアの蓄積メカニズム

新しいエージェントは最初、低い信頼スコアから始まります。成功を重ねるごとにスコアが上昇し、自律権限が拡大します。

```yaml
trust_escalation:
  new_agent:
    initial_trust: 0.3
    decision_level: CONFIRM  # 全て人間承認

  after_30_days:
    trust_threshold: 0.6
    decision_level: NOTIFY   # 実行後通知
    condition: "成功率 >= 90%"

  after_90_days:
    trust_threshold: 0.85
    decision_level: AUTO     # 自律実行
    condition: "成功率 >= 95% かつ HALT発動 0回"
```

## 6. 監査証跡はどう確保するのか？

### 全行動のログ記録

AEGISでは、エージェント間の全通信が記録されます。

```python
# services/api/app/core/audit.py
async def log_agent_action(
    agent_name: str,
    action_type: str,
    input_data: dict,
    output_data: dict,
    decision_level: str,
    confidence: float,
    tenant_id: str,
):
    """エージェントの行動を監査ログに記録"""
    await db.execute(
        audit_log.insert().values(
            agent_name=agent_name,
            action_type=action_type,
            input_hash=hash_data(input_data),
            output_hash=hash_data(output_data),
            decision_level=decision_level,
            confidence=confidence,
            tenant_id=tenant_id,
            timestamp=datetime.utcnow(),
            correlation_id=get_correlation_id(),
        )
    )
```

### なぜ監査証跡が重要なのか？

1. **問題の追跡**: 品質問題が発生した時、原因を特定できる
2. **パターンの発見**: 繰り返し失敗するパターンを検出し、予防策を講じられる
3. **コンプライアンス**: EU AI Act等の規制対応で必要になる
4. **信頼の構築**: エージェントの判断過程が透明であれば、人間の信頼を得やすい

## 7. 実運用から学んだ教訓とは？

### 教訓1: ガバナンスは最初から入れる

「まず動くものを作って、ガバナンスは後から」は最悪のアプローチです。ガバナンスなしで動き始めたエージェントに後からルールを適用すると、既存の動作との衝突で予期しない障害が発生します。

### 教訓2: 過度なガバナンスは機能しない

最初は全てをCONFIRM（人間承認）にしていました。結果として：
- 1日に50件以上の承認依頼が溜まる
- 承認疲れで内容を確認せずに承認し始める
- エージェントの処理が承認待ちでブロックされ続ける

**AUTO:NOTIFY:CONFIRM:HALTの理想的な比率は 60:25:12:3** です。

### 教訓3: 失敗を前提に設計する

エージェントは必ず間違えます。重要なのは「間違えないようにする」ことではなく、「間違えた時に被害を最小化し、素早く回復する」ことです。

```yaml
failure_isolation:
  blast_radius: "single_agent"     # 1体の失敗が全体に波及しない
  rollback: "automatic"             # 失敗時は自動ロールバック
  circuit_breaker:
    threshold: 3                    # 3回連続失敗でブレーカー発動
    reset_timeout: "5min"           # 5分後に再試行
  degradation: "graceful"           # 機能縮退で継続運用
```

### 教訓4: 3amテスト

「深夜3時にこのエージェントが暴走したら、人間が介入しなくても安全か？」——このテストに通らない設計は本番に出してはいけません。

AEGISでは、全てのエージェントが以下を満たします：
- 自動停止条件が設定されている
- 影響範囲が限定されている（blast radius）
- 失敗時の縮退運用が定義されている
- 監査ログが24時間以上保持される

## 8. まとめ — AIエージェントガバナンスの必須チェックリスト

- [ ] 4段階意思決定（AUTO/NOTIFY/CONFIRM/HALT）が定義されている

- [ ] 信頼度に基づくエスカレーションルールがある
- [ ] 全行動の監査証跡が記録されている
- [ ] 失敗時のサーキットブレーカーが設定されている
- [ ] 3amテストに合格している
- [ ] AUTO:NOTIFY:CONFIRM:HALTの比率が60:25:12:3に近い

ガバナンスは「AIの自由を奪うもの」ではありません。**AIが安全に自由を発揮するための基盤**です。24体のAIエージェントが同時に動く環境で学んだこの教訓が、あなたのAIエージェント運用の参考になれば幸いです。
